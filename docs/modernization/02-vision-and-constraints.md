# 02 — Vision & Constraints

Phase 2 output of the [modernization workflow](README.md). Captures *what success looks like* so phase 3 (ADRs) has evaluation criteria and phase 4 (decomposition) has prioritization signal. Strategic anchors and sacred cows are already locked in [`00-seed.md`](00-seed.md); they are referenced but not re-asked here.

> **Primary directives** ([ADR 0022](adrs/0022-modernization-primary-directives.md)) trump every other guidance in this workflow:
>
> 1. **Working software over comprehensive documentation.** Skills / agents / story bodies serve shipping behavior — they're not deliverables in their own right. Doc drift is a nudge, not a blocker.
> 2. **Business logic in the DDD domain, not the database.** The schema enforces what *must* be invariant for data to parse (PKs, FKs, structural NOT NULL, partial UNIQUE for identity, indexes). State machines, ranges, calculations, and other business rules live on aggregates as Java methods.
>
> When a section below reads as more prescription than working-software value: the directives win, the section yields.

Generated by `/modernize-vision`. Elicited interactively from the operator on 2026-05-14; **amended 2026-05-15 (a)** with end-user improvement scope (6 new features pre-cutover; timeline extended ~2 months); **amended 2026-05-15 (b)** with the mobile-first + airfield-efficiency UX directive (mobile-first across the whole app; dense-desktop variant for the flight-edit form; flight-edit + flight-list designated as the airfield hot-path screens); **amended 2026-05-17 (c)** with the self-service migration path + freemium SaaS model (AlpenFlight repositioned as a multi-tenant SaaS that any legacy FLS deployment can self-onboard onto: public landing → Keycloak signup → sandbox demo → legacy export JAR → encrypted upload → 72 h trial → subscribe or auto-delete). Amendment additions are tagged inline as `(2026-05-15a)`, `(2026-05-15b)`, or `(2026-05-17c)`.

## 4. Soft preferences

Bias the ADRs but aren't go/no-go. Phase 3 should call these out when an option clearly violates one.

- **Boring tech.** Solo operator on a 6–12 month timeline. Prefer mature, well-documented, easy-to-hire-for stacks over novel ones.
- **Type sharing between backend and frontend.** A single source of truth for DTOs / enums (codegen from OpenAPI, shared schema, or shared TS types via a JVM-to-TS bridge) — addresses R5 directly.
- **Fewer moving parts.** Solo ops favors fewer services, fewer queues, fewer infra components. Don't introduce a message broker, a separate worker tier, or a microservices split unless an ADR proves the value.
- **Self-hosted on a VPS / single host.** Cloud-managed services (RDS, Cloud Run, etc.) are fine but not preferred; vendor lock-in is what we're escaping. Containerization is welcome; Kubernetes is not required.
- **Reproducible local dev.** Docker Compose to bring up DB + dependencies; one-command `start` for the app. Current setup needs a long booby-trap list (see [`fls-e2e-setup.md`](../../home/agent/.claude/projects/-c-Users-roman-IdeaProjects-fls/memory/fls-e2e-setup.md) memory) — the new system should not.
- **TCO matters as a tiebreaker.** When two ADR options are close, pick the cheaper-to-run one (O5).
- **OGN ingestion endpoint should look like a normal API.** If a POST endpoint replaces direct-DB writes, it should be REST-shaped, authenticated, validated — not just a thin DB-write shim.
- **Informal success measurement for end-user improvements.** *(2026-05-15a amendment.)* The 6 amendment features (C17) are measured via feature-usage telemetry from the observability stack + operator gut-check at 3 months post-go-live. No formal adoption thresholds; if the operator decides a feature flopped, decommission is a normal product-management decision, not a vision-doc gate.
- **Inline validation, not on-blur.** *(2026-05-15b.)* Legacy validates on blur and surfaces errors in a top-level message bar. The new flight-edit form should validate inline (as the user types or moves focus) and pin errors next to the offending field. Reduces save-cycle latency; surfaces problems before the user reaches the bottom of a tall form.
- **Native input types over custom controls.** *(2026-05-15b.)* Where mobile browsers offer good native input (`type="time"`, `type="date"`, `inputmode="numeric"`, `autocomplete` hints), prefer them over JavaScript widgets. Keeps the form fast, accessible, and resilient when JS is slow on a phone. Legacy uses text inputs with format-on-blur — we should not carry that forward.
- **Recency bias in autocompletes.** *(2026-05-15b.)* Aircraft / pilot / location / route dropdowns surface "recently used" (per-user, last 7 days) at the top of the list before the rest of the catalog. Legacy already has localStorage-backed "Copy from Last"; recency bias generalizes that pattern to all autocompletes.
- **Aggressive prefetch for hot-path screens.** *(2026-05-15b.)* On app start (or login), prefetch the catalogs the flight-edit form needs (aircraft for the club, persons, locations, flight-types, routes) so the form opens with data already in the Signal Store. Worth a few hundred KB of upfront bandwidth to make the form instant.
- **Optimistic-concurrency UX must not lose user edits.** *(2026-05-15b.)* When a 409 conflict surfaces from the server (per S-067), the dense-desktop and mobile flight-edit forms show the conflict diff inline, keep the user's draft visible, and let them choose field-by-field merge or "keep mine / keep theirs / cancel". Aligned with C18 PWA conflict semantics but applies online too.
- **Stripe is the assumed billing provider until ADR 0021 says otherwise.** *(2026-05-17c.)* Mature, well-documented, hosted checkout + customer portal eliminate PCI scope, native EU/CH residency via Stripe's EU entity (Stripe Payments Europe Ltd., Ireland). Alternatives worth comparing in ADR 0021: Paddle (handles VAT-MOSS for free), Adyen (heavier integration, lower fees at scale).
- **Single-file Java JAR for the legacy export tool — same JVM as the server.** *(2026-05-17c.)* Operator already maintains a Java skill (C2); shipping the export tool on the same JVM avoids a second language. Compiled with GraalVM native-image is a *possible* later upgrade if startup time hurts, but not for v1 — JIT cold-start cost is dwarfed by the SQL-Server-read cost anyway.
- **Funnel telemetry from signup onwards.** *(2026-05-17c.)* Each step in the migration path (landing-view, signup-start, signup-complete, demo-start, jar-download, upload-start, upload-complete, trial-active, subscribed, deleted) emits a structured event into the observability stack. The operator must be able to ask "where did this user drop off?" without reading logs by hand. Adoption metric for O8 depends on it.
- **Sandbox demo seed should look real, not Lorem-Ipsum.** *(2026-05-17c.)* The sandbox tenant's seeded data uses plausible Swiss club names, aircraft registrations matching CH-airframe patterns, and realistic flight durations. A prospective admin evaluating the product should see something that looks like their world, not test data. Synthetic, but believable.
- **Free tier should be useful, not crippled.** *(2026-05-17c.)* The freemium gates per C30 are deliberately set so a tiny one-aircraft club can run AlpenFlight for free *if they accept the limits*. The pitch is not "trial-ware" — it's "we hope you stay free if you're a 5-pilot weekend club, and we hope you upgrade if you're a 50-pilot operation". This is also a deliberate moat against re-self-hosting legacy FLS for free: if the free tier covers small clubs adequately, they have no reason to maintain their own server.

## 5. Migration UX (per-tenant onboarding)

AlpenFlight is a self-service SaaS — there is **no centralized cutover event**. Each legacy FLS deployment migrates whenever its admin decides, via the same export-tool + UI-upload flow. The operator's own clubs are early customers of this flow, not a special case.

What the migration path must respect:

- **End-to-end self-service.** From "first hit on alpenflight.ch" to "logged in viewing own data in trial tenant" must be possible without operator intervention, in ≤ 30 minutes for a typical legacy DB (per O8).
- **Export-tool UX.** The Java JAR (S-139) is the only thing the admin runs by hand. CLI options must be obvious; errors must be diagnosable from stderr alone; the README must walk through a worked example end-to-end.
- **Upload-UI UX.** The browser upload (S-141) must show progress per phase (upload → decrypt → ingest → provision), surface clear errors, and let the admin retry from the failed step without re-running the JAR if possible.
- **CI is the rehearsal.** The S-016 schema-mapping library + parity oracle exercises the full JAR → upload → ingest path on every push against a seeded legacy SQL Server fixture. No calendar-bound rehearsal events.
- **Parity oracle.** Row counts, FK integrity, sampled-value diffs (S-016) gate every change to the mapping library. The legacy regression corpus (`DeliveryCreationTest`, Playwright depth specs S-101..S-106) runs against the new stack continuously, not just before a cutover.
- **Per-tenant failure mode.** A failed ingest affects one tenant. There is no "rollback plan" because there is no synchronized switchover; if a tenant's ingest fails, that tenant simply isn't provisioned and the admin tries again with a fresh handshake.
- **Auth.** Legacy passwords are never migrated. The signing-up admin uses Keycloak (S-134); other legacy users on the same FLS deployment become Postgres records without Keycloak identities until invited (post-subscription feature, future story) — or bulk-provisioned by the operator for their own clubs via S-028.
- **OGN / Proffix handoff.** Per-tenant, asynchronous, on each tenant's own schedule (S-149 / S-150). Not coordinated globally.
- **Operator's own clubs.** Same flow as every other customer. Operator-owned tenants get manually flipped to `active` (skipping `trial`) via an admin endpoint (S-028 / S-137) once their bundle is ingested.

## 6. Decision criteria for phase 3

Ordered list. Phase 3 uses this ranking to break ties between ADR options. Higher = more weight.

1. **Off-EOL & long-supported.** The option uses runtime versions with vendor support extending well past production go-live. (O1, the primary outcome.)
2. **Team-familiar stack.** The option is in Java or Kotlin, or in an ecosystem the solo operator can be productive in within weeks rather than months. (C2.)
3. **Linux-first, Windows-free.** The option runs natively on Linux with no Windows-only dependencies. (C1.)
4. **Swiss / EU data residency compatible.** The option can be hosted entirely in Switzerland or the EU without forcing US-region services. (C4.)
5. **Structural multi-tenancy supported.** The option makes "every read goes through the tenant guard" the path of least resistance — preferably via an ORM/repository layer that won't let you skip it. (C3, sacred cow.)
6. **Preserves sacred cows cleanly.** The option doesn't fight against the two-dimensional flight state, time gates, user/person split, rules-engine semantics, OGN/Proffix contracts. (C7.)
7. **Solo-operator operability.** The option doesn't require running a Kafka cluster, a Kubernetes control plane, or three separate services to deliver one user request. Single-binary or compose-up is preferred. (Soft, but high-weight given team shape.)
8. **Enables fast feature dev on the new stack.** The option has good codegen / scaffolding / shared-type-between-backend-and-frontend story. (O2, R5.)
9. **Has a credible migration story for our shape of data.** The option's migration tooling can replay our hand-rolled SQL history (R7) into a new schema and validate parity. (C9, C11.)
10. **Lower TCO.** Between near-equivalent options, pick the cheaper to run. (O5.)
11. **Mature ecosystem for our integration points.** OAuth / OIDC libs, Excel export, OpenAPI codegen, email templating, observability all available as well-maintained packages. (Soft.)
12. **Supports the C17 end-user improvements within the chosen stack.** *(2026-05-15a amendment.)* PWA + service worker + IndexedDB queue (for offline writes) + Web Push subscription (for notifications) + per-tenant CSS-variable theming (for whitelabel) — the chosen stack must accommodate these without architectural contortions. Re-evaluation of existing ADRs at this rank: Angular 21 (ADR 0004) already supports PWA + signals + CSS variables cleanly; Spring Boot 4.x (ADR 0001) accommodates Web Push delivery via standard libraries; Postgres 17 (ADR 0002) handles whitelabel asset bytea. **No existing ADR fails on this criterion.** Listed for visibility to phase 3 ADR review and to bias tie-breaking on the new ADRs implied by §8.
13. **Supports a single-component responsive mobile-first model with a dense-desktop variant.** *(2026-05-15b amendment.)* The chosen frontend stack must make CSS-driven density (Tailwind utility breakpoints, container queries, CSS variables) the cheap path; parallel mobile / desktop component trees must be discouraged by convention. Re-evaluation of existing ADRs at this rank: Angular 21 + Tailwind 4 (ADR 0004) supports container queries + variant-based density cleanly; NgRx Signal Store (ADR 0006) is layout-agnostic and binds equally to both density modes. **No existing ADR fails on this criterion.** This criterion is also the input to a new ADR 0017 (responsive breakpoint + density conventions) implied by C21–C22 — see §8.
14. **Supports a self-service multi-tenant SaaS lifecycle.** *(2026-05-17c amendment.)* The auth stack (Keycloak per ADR 0007) must support signup-enabled flows with Google IdP federation; the tenancy stack (`@TenantId` on Club per ADR 0008) must accommodate runtime-provisioned Deployments (parent of 1..N Clubs) without schema-time changes; the scheduled-jobs stack must iterate Deployments by lifecycle state (`trial`, `active`, etc.) and skip `sandbox`. Re-evaluation: Keycloak supports both prerequisites natively; Hibernate `@TenantId` accommodates runtime-created Clubs underneath a runtime-created Deployment without schema changes; Spring `@Scheduled` + `UnscopedTenantContext` (S-023) wraps the `DeploymentContext` cleanly. **No existing ADR fails on this criterion.** New ADRs 0018–0021 derive from this criterion. See §8.
15. **Migration-tool runtime parity with the server stack.** *(2026-05-17c amendment.)* The legacy export tool (C28) is a single-file Java JAR — same JVM as the server (ADR 0001). Avoids forcing the operator to maintain a second language for the migration tooling; lets the server-side bundle-schema parser and the client-side bundle-schema writer share types via a small `alpenflight/migration-bundle/` module consumed by both `alpenflight/server/` and the JAR build.

## 7. What is NOT a constraint here

Recording explicitly so phase 3 doesn't assume otherwise:

- **No specific backend framework chosen.** Java vs. Kotlin and Spring Boot vs. Micronaut vs. Quarkus vs. Ktor is an open ADR. C2 narrows the language family; the framework is still up for picking.
- **No specific frontend framework chosen.** The current is AngularJS 1.4; the replacement is open. React / Vue / Svelte / SolidJS all on the table.
- **No specific database engine chosen.** Current is SQL Server; reshape is allowed (C9); engine swap is an open ADR. Postgres is the obvious default given Linux + Swiss-hosting compatibility, but it's an ADR not a constraint.
- **No specific API shape locked in.** REST is the incumbent; tRPC / GraphQL / OpenAPI-codegen are all open.
- **No specific background-job mechanism locked in.** Current is "OS cron → HTTP endpoint → in-process dispatch". The new system might keep that shape, switch to in-process Quartz, or use a managed scheduler. Open ADR.

### Phase-3 ADR candidates implied by the 2026-05-17c amendment

The self-service migration + freemium SaaS scope (C25–C33) introduces four cross-cutting decisions. The marketing landing copy, sandbox seed data, and per-feature UI gates are story-level — the architectural shape is ADR-worthy.

- **ADR 0018 — Self-service Deployment lifecycle & provisioning.** Locks the new `Deployment` entity (above Club; FK `Club.deployment_id`), the `Deployment.lifecycle_state` enum (`sandbox / trial / active / past_due / cancelled / deleting`), the transition rules (signup → `trial` only after first successful ingest, not on signup; subscription webhook → `active`; etc.), the provisioning code path (Keycloak group → Deployment row → 1..N Club rows from the bundle → admin user → seed default reference data per Club), how scheduled jobs filter by state (jobs skip `sandbox` and `deleting`; some skip `trial`), and how the `DeploymentContext` enumerates child Clubs for cross-cutting reads (bulk-provision, trial-delete cascade, freemium caps). - **ADR 0019 — Legacy FLS export-bundle format + encryption protocol.** Locks the bundle archive shape (gzip-tar with manifest + per-entity NDJSON streams + a SHA-256 manifest), the schema-version field, the encryption scheme (RSA-4096 server-generated keypair per upload session; bundle payload encrypted with a one-time AES-256-GCM session key; AES key wrapped under the RSA public key — i.e. hybrid encryption), the JAR's CLI contract (`--jdbc-url`, `--user`, `--password`, `--public-key-file`, `--output`), and the schema parity strategy (re-exportable from a seeded legacy DB per memory `[[feedback-re-runnable-over-frozen-docs]]`, *not* a committed JSON snapshot). Cross-cutting because the JAR build, the upload endpoint, the decrypt-and-ingest pipeline, and the parity test corpus all share the format.

Cross-cutting because every gated feature inherits the same annotation + response shape; without ADR the convention drifts per epic.

Locks the integration shape (hosted checkout + customer portal; webhook → `Club.lifecycle_state` transition; ID-only storage; PCI scope kept outside AlpenFlight per C33). Decision unblocks the billing stories.

Marketing landing copy, sandbox seed data, JAR CLI ergonomics, and freemium-tier UI affordances are story-level. The export bundle's *schema* (i.e. which entities go in the bundle) is story-level (re-derive from legacy `flsserver/` and the existing `S-010` parity baseline); only the *format* is ADR-locked.
