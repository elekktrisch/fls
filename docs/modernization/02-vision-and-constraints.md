## 4. Soft preferences

Bias the ADRs but aren't go/no-go. Phase 3 should call these out when an option clearly violates one.

- **Boring tech.** Solo operator on a 6–12 month timeline. Prefer mature, well-documented, easy-to-hire-for stacks over novel ones.
- **Type sharing between backend and frontend.** A single source of truth for DTOs / enums (codegen from OpenAPI, shared schema, or shared TS types via a JVM-to-TS bridge) — addresses R5 directly.
- **Fewer moving parts.** Solo ops favors fewer services, fewer queues, fewer infra components. Don't introduce a message broker, a separate worker tier, or a microservices split unless an ADR proves the value.
- **Self-hosted on a VPS / single host.** Cloud-managed services (RDS, Cloud Run, etc.) are fine but not preferred; vendor lock-in is what we're escaping. Containerization is welcome; Kubernetes is not required.
- **Reproducible local dev.** Docker Compose to bring up DB + dependencies; one-command `start` for the app. Current setup needs a long booby-trap list (see [`fls-e2e-setup.md`](../../home/agent/.claude/projects/-c-Users-roman-IdeaProjects-fls/memory/fls-e2e-setup.md) memory) — the new system should not.
- **TCO matters as a tiebreaker.** When two ADR options are close, pick the cheaper-to-run one (O5).
- **OGN ingestion endpoint should look like a normal API.** If a POST endpoint replaces direct-DB writes, it should be REST-shaped, authenticated, validated — not just a thin DB-write shim.
- **Informal success measurement for end-user improvements.** *(2026-05-15a amendment.)* The 6 amendment features (C17) are measured via feature-usage telemetry from the observability stack + operator gut-check at 3 months. No formal adoption thresholds; if the operator decides post-cutover that a feature flopped, decommission is a normal product-management decision, not a vision-doc gate.
- **Inline validation, not on-blur.** *(2026-05-15b.)* Legacy validates on blur and surfaces errors in a top-level message bar. The new flight-edit form should validate inline (as the user types or moves focus) and pin errors next to the offending field. Reduces save-cycle latency; surfaces problems before the user reaches the bottom of a tall form.
- **Native input types over custom controls.** *(2026-05-15b.)* Where mobile browsers offer good native input (`type="time"`, `type="date"`, `inputmode="numeric"`, `autocomplete` hints), prefer them over JavaScript widgets. Keeps the form fast, accessible, and resilient when JS is slow on a phone. Legacy uses text inputs with format-on-blur — we should not carry that forward.
- **Recency bias in autocompletes.** *(2026-05-15b.)* Aircraft / pilot / location / route dropdowns surface "recently used" (per-user, last 7 days) at the top of the list before the rest of the catalog. Legacy already has localStorage-backed "Copy from Last"; recency bias generalizes that pattern to all autocompletes.
- **Aggressive prefetch for hot-path screens.** *(2026-05-15b.)* On app start (or login), prefetch the catalogs the flight-edit form needs (aircraft for the club, persons, locations, flight-types, routes) so the form opens with data already in the Signal Store. Worth a few hundred KB of upfront bandwidth to make the form instant.
- **Optimistic-concurrency UX must not lose user edits.** *(2026-05-15b.)* When a 409 conflict surfaces from the server (per S-067), the dense-desktop and mobile flight-edit forms show the conflict diff inline, keep the user's draft visible, and let them choose field-by-field merge or "keep mine / keep theirs / cancel". Aligned with C18 PWA conflict semantics but applies online too.

## 5. Cutover plan shape

Not a design — just the constraints any cutover design must respect.

- **Downtime budget:** ≤ 6 hours, overnight (start ~22:00 local, finish before 06:00). Pick a weekday in a low-traffic season (mid-week, late autumn / winter).
- **Parity-validation strategy:** expand `DeliveryCreationTest` corpus to ≥ 1 case per production rule-filter combination per club + expand the Playwright suite to cover state-machine guards, time-gate boundaries, rules-engine combinations, and the validation rejection paths called out in R14. Run the corpus on both old and new systems and diff before cutover. *No* dual-run with shared DB writes (seed forbids it).
- **Rehearsal:** the migration script must run end-to-end against a production-data-shaped staging DB at least once, with verification automation that proves row counts, foreign-key integrity, and rules-engine output match. Two rehearsals preferred.
- **Rollback definition:** if cutover fails before users are let in, restore the old system from its DNS / proxy entry and the pre-cutover DB snapshot — i.e. the old system was never decommissioned, only had traffic stopped. Once users are let in to the new system and have written data, "rollback" means staying on the new system and forward-fixing.
- **Auth migration:** passwords are NOT migrated (C14). On cutover, all users get an email with a reset link. The cutover script must export the user→email list from the old DB and queue the resets.
- **OGN handoff (C8):** OGNAnalyser must be flipped from direct-DB writes to the new ingestion API at the cutover instant, in lockstep. Requires the maintainer's availability during the window — book this early.
- **Proffix handoff:** depends on §8 investigation (where the live code actually lives). New system must serve a Proffix-compatible API on cutover day.
- **Feature-scope additions per C17 (2026-05-15 amendment).** All 6 amendment features ship pre-cutover. Cutover-day verification gates expand: each feature has a smoke check (PWA service worker installs + queues a write offline + syncs on reconnect; push notification subscribes + delivers a test message; whitelabel renders correctly for the test club; iCal feed produces a parseable calendar; waitlist enrolls + notifies on cancellation; reservation conflict UI highlights the right rows). The **downtime budget remains ≤ 6 hr** — amendment features add to development time, not cutover-day execution time. Timeline extension (~2 months) absorbed by pushing the rehearsal + cutover dates, not by compressing the cutover window itself.

## 6. Decision criteria for phase 3

Ordered list. Phase 3 uses this ranking to break ties between ADR options. Higher = more weight.

1. **Off-EOL & long-supported.** The option uses runtime versions with vendor support extending well past cutover. (O1, the primary outcome.)
2. **Team-familiar stack.** The option is in Java or Kotlin, or in an ecosystem the solo operator can be productive in within weeks rather than months. (C2.)
3. **Linux-first, Windows-free.** The option runs natively on Linux with no Windows-only dependencies. (C1.)
4. **Swiss / EU data residency compatible.** The option can be hosted entirely in Switzerland or the EU without forcing US-region services. (C4.)
5. **Structural multi-tenancy supported.** The option makes "every read goes through the tenant guard" the path of least resistance — preferably via an ORM/repository layer that won't let you skip it. (C3, sacred cow.)
6. **Preserves sacred cows cleanly.** The option doesn't fight against the two-dimensional flight state, time gates, user/person split, rules-engine semantics, OGN/Proffix contracts. (C7.)
7. **Solo-operator operability.** The option doesn't require running a Kafka cluster, a Kubernetes control plane, or three separate services to deliver one user request. Single-binary or compose-up is preferred. (Soft, but high-weight given team shape.)
8. **Enables fast feature dev post-cutover.** The option has good codegen / scaffolding / shared-type-between-backend-and-frontend story. (O2, R5.)
9. **Has a credible migration story for our shape of data.** The option's migration tooling can replay our hand-rolled SQL history (R7) into a new schema and validate parity. (C9, C11.)
10. **Lower TCO.** Between near-equivalent options, pick the cheaper to run. (O5.)
11. **Mature ecosystem for our integration points.** OAuth / OIDC libs, Excel export, OpenAPI codegen, email templating, observability all available as well-maintained packages. (Soft.)
12. **Supports the C17 end-user improvements within the chosen stack.** *(2026-05-15a amendment.)* PWA + service worker + IndexedDB queue (for offline writes) + Web Push subscription (for notifications) + per-tenant CSS-variable theming (for whitelabel) — the chosen stack must accommodate these without architectural contortions. Re-evaluation of existing ADRs at this rank: Angular 21 (ADR 0004) already supports PWA + signals + CSS variables cleanly; Spring Boot 4.x (ADR 0001) accommodates Web Push delivery via standard libraries; Postgres 17 (ADR 0002) handles whitelabel asset bytea. **No existing ADR fails on this criterion.** Listed for visibility to phase 3 ADR review and to bias tie-breaking on the new ADRs implied by §8.
13. **Supports a single-component responsive mobile-first model with a dense-desktop variant.** *(2026-05-15b amendment.)* The chosen frontend stack must make CSS-driven density (Tailwind utility breakpoints, container queries, CSS variables) the cheap path; parallel mobile / desktop component trees must be discouraged by convention. Re-evaluation of existing ADRs at this rank: Angular 21 + Tailwind 4 (ADR 0004) supports container queries + variant-based density cleanly; NgRx Signal Store (ADR 0006) is layout-agnostic and binds equally to both density modes. **No existing ADR fails on this criterion.** This criterion is also the input to a new ADR 0017 (responsive breakpoint + density conventions) implied by C21–C22 — see §8.

## 7. What is NOT a constraint here

Recording explicitly so phase 3 doesn't assume otherwise:

- **No specific backend framework chosen.** Java vs. Kotlin and Spring Boot vs. Micronaut vs. Quarkus vs. Ktor is an open ADR. C2 narrows the language family; the framework is still up for picking.
- **No specific frontend framework chosen.** The current is AngularJS 1.4; the replacement is open. React / Vue / Svelte / SolidJS all on the table.
- **No specific database engine chosen.** Current is SQL Server; reshape is allowed (C9); engine swap is an open ADR. Postgres is the obvious default given Linux + Swiss-hosting compatibility, but it's an ADR not a constraint.
- **No specific API shape locked in.** REST is the incumbent; tRPC / GraphQL / OpenAPI-codegen are all open.
- **No specific background-job mechanism locked in.** Current is "OS cron → HTTP endpoint → in-process dispatch". The new system might keep that shape, switch to in-process Quartz, or use a managed scheduler. Open ADR.

### Phase-3 ADR candidates implied by the 2026-05-15a amendment

The amendment introduces three architectural decisions that warrant dedicated ADRs (operator chose "ADRs for cross-cutting decisions; stories for everything else"):

- **ADR 0014 — PWA offline-write architecture.** Service worker + IndexedDB queue schema + `sync` event trigger semantics + reconciliation-prompt UX flow + the server-side handle for conflict detection (e.g. ETag / `If-Match` / version column — already plumbed by S-067). Cross-cutting because it touches every mutating endpoint and shapes the Signal Store cache invalidation pattern (S-006).
- **ADR 0015 — Push notification provider.** Web Push (VAPID) — built-in browser support, no third-party — vs. Firebase Cloud Messaging (broader device support, Google dependency) vs. third-party (OneSignal, Pusher). Recommend Web Push as the boring choice; ADR forces the comparison.
- **ADR 0016 — Per-tenant theming mechanism.** Runtime CSS-variable injection vs. build-time theming. Recommend runtime CSS variables fetched from `/api/v1/clubs/myClub/branding` with cache; ADR locks the convention so future tenant-customizable surfaces follow it.

Whitelabel asset storage location is already decided (Postgres bytea, per amendment §2 NFR) — no separate ADR required.

In-app inbox is a domain feature (entity + CRUD endpoints + UI); story-level under masterdata or a dedicated `notifications/` module. No ADR.

Reservation enhancements (waitlist, iCal, conflict-detection UI) are story-level under the existing E-07 / reservations epic. No ADR.

### Phase-3 ADR candidates implied by the 2026-05-15b amendment

The mobile-first + dense-desktop directive (C21–C24) introduces one cross-cutting decision that warrants its own ADR. The flight-edit form itself is a story-level concern (S-062c) — the directive shapes it but doesn't promote it to ADR.

- **ADR 0017 — Responsive breakpoint + density conventions.** Locks the breakpoint set (`sm` 360, `md` 768, `lg` 1024, `xl` 1440), the layout grid (Tailwind utilities + container queries for component-local density), the density-toggle convention (CSS-class-driven, no parallel component trees per C22), and the touch-target rule (≥ 44 px on `<lg`, ≥ 28 px on `≥lg` for secondary actions per §2 NFR). Cross-cutting because every primitive in the component kit (S-008) and every form (S-007) inherits from it; deserves a single decision instead of being re-litigated per story.

### Flight-edit form — recommended optimization recipe (2026-05-15b)

Drawn from the legacy form analysis + operator's "manual-copy stays, empty-state UX must also be excellent" pushback. Targets S-062b (list) + S-062c (form); stories under the airfield hot-path (C23) inherit these conventions. **This is a recipe, not a contract** — the implementing stories tailor it; the listed items below are what good looks like.

| # | Technique | Why it pays off | Where it lands |
|---|---|---|---|
| F1 | **Single responsive component, CSS-driven density** (`@container` + Tailwind variant) | Avoids the desktop/mobile branch the legacy AngularJS code never even attempted; one logic surface, two layouts. | C22; S-062c |
| F2 | **Mobile layout: single-column, sectioned accordion** (Aircraft → Pilots → Times → Locations → Costs) with a sticky save bar at the bottom | A 35-field form on a phone is a height monster; sections + sticky save keep the user oriented and one tap from "done". | S-062c (mobile breakpoint) |
| F3 | **Desktop dense layout: 4-column grid, inline labels, tight padding** with glider + tow side-by-side (mirroring legacy intent, but tighter) | Operator's existing mental model; keyboard-only completion (NFR row) requires a tight tab path. | S-062c (lg+ breakpoint) |
| F4 | **Keyboard shortcuts on dense: Enter = save; Ctrl+D = save+copy; Esc = cancel-with-confirm; 1–5 = common flight-type select; Tab/Shift+Tab natural-order traversal** | The operator's literal ask: "avoid too much mouse clicks". | S-062c |
| F5 | **"Copy from Last" buttons preserved** on aircraft / tow aircraft / routes / engine counters / cost-balance, backed by `localStorage` per-aircraft (matches legacy keys conceptually) | Operator's explicit pushback (C24). Replaces nothing; lives alongside (F6). | S-062c |
| F6 | **Smart defaults from server context** via a `GET /api/v1/flights/last-context?aircraftId=...&pilotId=...&date=...` hint endpoint that returns the last-saved field combo for the same aircraft + day; applied only when the form opens blank | Empty-state UX for the first flight of the day — when "Copy from Last" has nothing to copy from. | S-062a (endpoint) + S-062c (consumer) |
| F7 | **Per-club default flight-type + cost-balance type + start location**, fetched once per session into the Signal Store | First-flight-of-day still has sensible defaults even when (F6) returns nothing. | S-047 (reference data) + S-062c |
| F8 | **Recency-biased autocomplete dropdowns** (aircraft, pilot, observer, passenger, location, route) — recent 7-day picks float to top, rest of catalog below | Generalizes the legacy localStorage trick to all dropdowns. Mobile-friendly: less scrolling. | S-008 (primitive: `<fls-autocomplete>`) + S-062c |
| F9 | **Inline validation, not on-blur** (soft pref §4); errors pin next to the field, not in a top message bar | Catches errors before the user scrolls to the save button on mobile. | S-007 (forms convention) + S-062c |
| F10 | **Native input types** (`type="time"`, `type="date"`, `inputmode="numeric"`, `autocomplete` hints) instead of text-with-format-on-blur | Faster, accessible, gives mobile the OS-native time picker. Legacy doesn't do this. | S-007 + S-062c |
| F11 | **Aggressive prefetch on app start** of aircraft / persons / locations / flight-types / routes into the Signal Store; flight-edit form opens with data in-hand | Form-open feels instant; supports the time-to-log NFR. | S-006 (Signal Store reference) + S-062c |
| F12 | **Auto-save draft to IndexedDB on every field change** (debounced 500 ms); on connection loss, queue via PWA service worker (C17 / C18) | Form-state survives navigation, reload, crash, connection drop. Legacy users currently lose entries when reception flaps. | ADR 0014 (PWA architecture) + S-062c |
| F13 | **Conflict-resolution UX inline in the form** when a `409 If-Match` fails on save: show diff with per-field "keep mine / keep theirs", form stays open, no draft loss | Online application of C18 PWA conflict semantics; protects partial entries. | S-067 (concurrency) + S-062c |
| F14 | **`<fls-time-now-button>` primitive** — replaces legacy "Set Now" pattern; sets `now` rounded to nearest minute; works one-handed on phone | Existing operator behavior, moved into a reusable primitive instead of inlined per-field. | S-008 + S-062c |
| F15 | **Conditional sections render-controlled by Signal Store**, not template `ng-if` cascades; dependent fields appear with a 150ms slide-in, focus moves to the first new field | Legacy AngularJS triggers layout jank when flight-type changes — bad mobile experience. Signal Store keeps it smooth. | S-006 + S-062c |
| F16 | **OGN landing-time hint (deferred / non-blocking)** — if an OGN ingest record matches the in-progress draft (same aircraft + same day), surface a "Auto-fill landing time from OGN?" toast; one tap fills | Reduces "Set Now" presses for ports with OGN coverage. Optional, not on the critical path; can land post-cutover. | S-066 (OGN) + S-062c (future iteration; flagged as `nice-to-have`) |

**Out of scope for this amendment:** bulk-entry / spreadsheet mode (deferred — can be added later if operator validates demand); voice input (deferred); native-app rewrap (not in scope — PWA-only per C17).
