# 0019 ‚Äî Entity ID strategy

- **Status:** Accepted
- **Date:** 2026-05-16
- **Decision criteria** (from [vision ¬ß6](../02-vision-and-constraints.md#6-decision-criteria-for-phase-3)):
  2. Team-familiar stack
  5. Structural multi-tenancy supported
  6. Preserves sacred cows cleanly
  7. Solo-operator operability
  8. Enables fast feature dev post-cutover
  9. Credible migration story

## Context

The legacy schema uses `uniqueidentifier` (GUID) PKs everywhere except a handful of int-keyed reference tables. The new system must commit to an identity strategy that (a) works automatically with Hibernate 7 / Spring Data JPA per [ADR 0001](0001-backend-language-and-framework.md), (b) supports the DDD-with-Aggregates domain model per [ADR 0018](0018-domain-model-ddd-aggregates.md) (strong-typed IDs as value objects, aggregate roots as identity carriers across bounded contexts), (c) lets the application ‚Äî not the database ‚Äî own ID generation per the operator's stated preference, and (d) doesn't fight `@TenantId` multi-tenancy per [ADR 0008](0008-multi-tenancy-mechanism.md) (which already noted "the `@TenantId` column is just a normal indexed `bigint` or UUID column" ‚Äî no schema-level contradiction either way).

The operator's input on this ADR explicitly named UUID generated by the Java application (not via DB sequence) with automated Hibernate integration. The earlier-considered carve-out ‚Äî BIGINT for non-aggregate-root internal entities ‚Äî was offered but rejected in favor of conceptual uniformity. This ADR locks **pure UUID v7 PKs across every entity in the system**, application-generated via Hibernate 7's `@UuidGenerator(style = TIME)`.

UUID v7 (time-ordered, [RFC 9562](https://datatracker.ietf.org/doc/rfc9562/)) is the key choice within the UUID family. v4 (random) scatters inserts across B-tree index pages and degrades insert throughput at scale; v7's leading 48-bit Unix-millis timestamp gives near-monotonic insertion order ‚Äî within an order of magnitude of BIGINT IDENTITY on insert performance, with the conceptual benefits of UUIDs (globally unique, app-generated, no DB roundtrip, opaque across bounded contexts).

## Options considered

### Option A ‚Äî Pure UUID v7 everywhere

- **Capabilities:**
  - Every entity PK is `UUID NOT NULL PRIMARY KEY` (Postgres native `uuid` type).
  - Generation: Hibernate 7 `@UuidGenerator(style = TIME)` on every `@Id` field. Time-ordered (UUID v6/v7 family) ‚Äî the JVM constructs the UUID at `entityManager.persist()` time before the INSERT statement is built. No DB sequence; no `RETURNING` round-trip; no ID-not-yet-known state.
  - Strong-typed ID value objects per [ADR 0018](0018-domain-model-ddd-aggregates.md): `record FlightId(UUID value) {}`, `record AircraftId(UUID value) {}`, etc. ‚Äî compile-time prevention of "passed the wrong ID type" bugs.
  - Cross-aggregate references store the target's UUID directly (no need to distinguish aggregate-root vs internal IDs).
  - JSON payloads carry UUIDs as 36-char canonical strings (`8-4-4-4-12`). Jackson + Spring Web handle conversion natively.
  - Postgres `uuid` type is 16 bytes (vs 8 for `bigint`); B-tree indexes ~2x larger; UUID v7's time-ordered prefix preserves locality so insert throughput stays close to IDENTITY.
- **Fit to criteria:** Criterion 2 ‚úì (Hibernate 7 `@UuidGenerator` is the canonical pattern; documented; one annotation per entity). Criterion 5 ‚úì (no contradiction with `@TenantId` from ADR 0008 ‚Äî tenant column is independent of PK type; can itself remain UUID or BIGINT per ADR 0008's "BIGINT or UUID" neutrality, see Follow-ups). Criterion 6 ‚úì (S-016 cutover maps each legacy Guid ‚Üí new UUID v7 with a remapping table; FK rewrites mechanical). Criterion 7 ‚úì (one ID type to learn; one annotation to remember; no carve-out rule for which entities get which type). Criterion 8 ‚úì (Spring Modulith + per-aggregate repositories operate on a uniform identity surface). Criterion 9 ‚úì (legacy GUIDs are 128-bit; new UUIDs are 128-bit; cutover is a per-row remap, not a type conversion).
- **Migration cost:** Schema-cost: every PK column becomes `uuid` (was `BIGINT GENERATED BY DEFAULT AS IDENTITY` in the speculative refinements of S-012/S-013/S-014 ‚Äî those refinements need updating). Code-cost: one `@UuidGenerator` annotation per entity at S-022. Cutover cost: the S-016 migration script generates new UUIDs and rewrites FKs through a `legacy_id_map (legacy_guid, new_uuid)` lookup table. Bulk INSERTs into a UUID-keyed Postgres 17 table are within a few percent of IDENTITY throughput per published benchmarks (Tomas Vondra / Postgres 17 measurements) when v7 is used.
- **Ecosystem risk:** low. Hibernate 7's `@UuidGenerator` ships in `hibernate-core` (no extra dependency). Postgres `uuid` type is native (no extension needed). Jackson handles UUID ‚Üî JSON natively. Spring Data JPA's `findById(UUID)` works out of the box.
- **Index footprint:** material but bounded. Worked example at 5-year horizon assuming legacy growth rates:
  - `flight` 5M rows √ó 16 bytes PK = 80 MB heap; +9 indexes √ó ~20 bytes/entry = ~900 MB index space (vs ~450 MB with BIGINT) ‚Äî **+450 MB delta**.
  - `flight_crew` 25M rows √ó 16 bytes PK = 400 MB heap; +3 indexes √ó ~24 bytes/entry = ~1.8 GB index (vs ~900 MB BIGINT) ‚Äî **+900 MB delta**.
  - `aircraft_operating_counter` ~2M rows = trivial.
  - `audit_log` (S-027) at high volume = potentially gigabytes ‚Äî flag for S-027 perf-engineering.
  - Aggregate impact: ~3-5 GB additional disk + RAM at 5-year scale. Inside the cost envelope for a self-hosted VPS (ADR 0010); not free, but not blocking.
- **Escape hatch:** if a specific high-volume table proves perf-critical (audit_log most likely), local degradation to a `BIGINT id` + `UUID public_id` shape is a per-table migration without disturbing neighbors.

### Option B ‚Äî Hybrid: UUID v7 for aggregate roots, BIGINT IDENTITY for internal entities + association tables

- **Capabilities:**
  - Aggregate roots ([ADR 0018](0018-domain-model-ddd-aggregates.md)) ‚Äî `Flight`, `Aircraft`, `Person`, `Club`, `User`, `Delivery`, `AccountingRuleFilter`, `AircraftReservation`, `PlanningDay`, `Location`, `FlightType`, `Article` ‚Äî get UUID v7.
  - Internal entities (`FlightCrew`, `AircraftAircraftState`, `AircraftOperatingCounter`, `PlanningDayAssignment`, `DeliveryItem`, `InOutboundPoint`, reference lookups) get BIGINT IDENTITY.
  - Cross-aggregate FKs are UUIDs; intra-aggregate FKs are BIGINTs.
- **Fit to criteria:** Criterion 7 ‚úó ‚Äî two ID types means a per-entity decision rule that contributors must remember; carve-out boundary drifts as new entities are added; the cognitive overhead is real for a solo operator.
- **Why not chosen:** operator explicitly preferred conceptual uniformity over the index-footprint savings. The savings (~3-5 GB at 5-year scale) are material but inside the cost envelope; not worth the per-entity classification burden.

### Option C ‚Äî Pure UUID v4 (random) everywhere

- **Capabilities:** same shape as Option A, but UUIDs are random.
- **Fit to criteria:** Criterion 8 ‚úó ‚Äî random UUIDs scatter inserts across B-tree pages, causing measurable insert throughput degradation at scale. Published benchmarks show 30-60% insert slowdown vs IDENTITY at the 10M-row mark; UUID v7's time-ordered prefix keeps the slowdown to single digits.
- **Why not chosen:** UUID v4 buys nothing over v7 (both are opaque to humans, both are globally unique) and costs measurable throughput at scale.

### Option D ‚Äî Pure BIGINT IDENTITY (DB-generated)

- **Capabilities:** every PK is `BIGINT GENERATED BY DEFAULT AS IDENTITY`. Was the implicit choice in the speculative refinements of S-012 / S-013 / S-014.
- **Fit to criteria:** all criteria pass on paper.
- **Why not chosen:** operator's explicit ADR-args preference is UUID-generated-by-Java, not DB sequence. Boundary-crossing identifiers (URLs, JSON payloads, audit logs, integration-event correlation IDs) benefit from UUID's globally-unique property; sequential integers leak ordinal information ("Delivery #1234 vs Delivery #1235 are sequential in time and likely consecutive in your data").

## Decision

Chosen: **Option A ‚Äî Pure UUID v7 everywhere, application-generated via Hibernate 7 `@UuidGenerator(style = TIME)`**. Decision driven by the operator's explicit ADR-args preference (UUIDs, Java-generated, automated Hibernate integration), criterion 7 (one ID type, one annotation, one mental model ‚Äî minimizes solo-operator cognitive load), and criterion 6 (UUID v7 preserves S-016 cutover parity with legacy GUIDs ‚Äî both are 128-bit; cutover is a remap, not a conversion). The carve-out for BIGINT internal-entity IDs was considered (Option B) and explicitly rejected in favor of conceptual uniformity; the ~3-5 GB additional 5-year index footprint is inside the cost envelope.

Strong-typed ID value objects (`record FlightId(UUID value) {}`, etc.) per [ADR 0018](0018-domain-model-ddd-aggregates.md) compose with this decision: the underlying primitive is `java.util.UUID`, but every API surface uses the typed wrapper. Compile-time prevention of ID-mistake bugs is preserved.

## Aggregate-prefix scheme

Aggregate roots from [ADR 0018](0018-domain-model-ddd-aggregates.md) carry a 3-letter prefix at every **external boundary** (REST API URLs + JSON payloads, structured logs, error messages, audit-log `target.id` fields, user-facing error UI). The **database storage stays pure `uuid`** ‚Äî the prefix is a presentation concern applied at the boundary, not a column type.

### Encoded form

Aggregate-root IDs serialize as `<prefix>_<crockford-base32-uuid-v7>`:

```
flt_0jwq8egfk0fn400084ferpf08a8
```

- 3-letter prefix + `_` separator + 26-character Crockford base32 encoding of the 128-bit UUID v7. Total length: **30 characters**.
- [Crockford base32](https://www.crockford.com/base32.html) (lowercase): alphabet omits `I / L / O / U` to eliminate read-aloud and OCR ambiguity. Optional check character not used.
- UUID v7's time-ordered 48-bit prefix is preserved in the encoding ‚Äî IDs created in the same second share a leading substring, which makes log-scrolling and time-bucketing visually obvious without parsing the timestamp.

### Internal entities ‚Äî no prefix

Internal entities (`FlightCrew`, `DeliveryItem`, `AircraftAircraftState`, `AircraftOperatingCounter`, `PlanningDayAssignment`, `InOutboundPoint`, every `*_unit_type` lookup) keep raw UUIDs without a prefix. Internal IDs are implementation details of the aggregate; they rarely cross boundaries. If an internal ID does surface (debug log, ops console), the canonical UUID form is correct. This aligns with DDD: aggregate roots are the identity surface that crosses contexts; internals don't.

### Implementation surface

- **Typed-ID value objects** (per [ADR 0018](0018-domain-model-ddd-aggregates.md)) own the prefix:
  ```java
  public record FlightId(UUID value) implements EntityId {
      public static final String PREFIX = "flt";
      public String toExternal() { return PREFIX + "_" + Base32Crockford.encode(value); }
      public static FlightId parse(String external) { /* prefix-check + decode */ }
  }
  ```
- **Jackson serializer/deserializer** registered at app startup converts `FlightId ‚Üî "flt_..."` at every JSON boundary. One global registration; per-type via the typed-ID interface.
- **Spring URL path converter** (`Converter<String, FlightId>`) parses path variables: `GET /api/v1/flights/{id}` accepts `flt_0jwq8egfk0fn400084ferpf08a8` and yields a `FlightId`.
- **Structured logging MDC** writes the prefixed form. Log fields holding aggregate-root IDs (`flight_id`, `actor.user_id`, etc.) carry the prefix; raw UUIDs never appear in log lines.
- **Hibernate persistence layer** sees only `UUID` (the inner value). `@AttributeConverter` may not even be necessary ‚Äî the typed-ID `record` can hold a public `UUID value()` accessor and Hibernate maps the wrapper directly via `@Embeddable` / `@Convert` (decided at S-022).
- **psql debugging helper**: ship a function `flt(text) RETURNS uuid` and a sibling per-prefix function in a `next/server/src/main/resources/db/migration/V<n>__id_codec_functions.sql` migration (post-baseline) for ad-hoc dev queries.

### Cutover (S-016) implications

Legacy GUIDs (`uniqueidentifier` in SQL Server) are stored bit-for-bit compatible with Postgres `uuid`. At cutover:

- Option (i): preserve every legacy GUID as-is. Audit-log searchability across the cutover boundary is direct. Insert-locality for the legacy rows is whatever it was (random v4-equivalent); going forward, new IDs are v7 with good locality. The mixed pattern is benign.
- Option (ii): regenerate every legacy GUID as UUID v7 and persist a `legacy_id_map (legacy_guid, new_uuid)` lookup. Better insert locality across the whole dataset; audit-log historical searches require the lookup. More cutover-script work.

**Default recommendation:** option (i) ‚Äî preserve legacy GUIDs. The insert-locality cost is bounded (legacy rows are only mutated when their state machine advances; new inserts dominate write traffic post-cutover). S-016 may revisit if profiling shows hot tables suffer measurable degradation.

## Consequences

- **Positive:**
  - One ID type across every entity, every API, every database column. No per-entity decision rule.
  - Application-generated: `flight.id` is known the instant `new Flight(...)` returns ‚Äî no `entityManager.persist()` round-trip required before publishing to a queue, logging the new ID, or returning it in an HTTP response.
  - UUIDs are URL-safe (no leading zeros, no embedded slashes) and globally unique. External integrations (Proffix sync per [C10](../02-vision-and-constraints.md#3-hard-constraints), OGN ingestion per [C8](../02-vision-and-constraints.md#3-hard-constraints), audit-log correlation, push-notification deep links) carry the same ID without ordinal leakage.
  - S-016 cutover parity-friendly: legacy GUIDs (16 bytes) ‚Üí new UUIDs (16 bytes) via a 1:1 remap. Schema diff is the column type rename (`uniqueidentifier` ‚Üí `uuid`); data preserved bit-for-bit if the remap chooses to keep legacy GUIDs as v4 strings instead of regenerating as v7. **Open Q at S-016:** preserve legacy GUIDs (preserves audit-log searchability into historical data) vs regenerate as v7 (better insert locality going forward; need a lookup table for historical references). Default recommendation: **regenerate as v7 + persist a `legacy_id_map (legacy_guid, new_uuid)` lookup** for historical correlation.
  - Strong-typed ID value objects compose: `FlightId`, `AircraftId`, `PersonId`, etc. wrap `UUID`; cross-entity passing is compile-time-checked.
  - Hibernate 7 `@UuidGenerator(style = TIME)` is one annotation per entity (or one shared `@MappedSuperclass` for the convention) ‚Äî no per-entity boilerplate.
  - Postgres 17 `uuid` is native; `gen_random_uuid()` is available for the rare migration-time INSERT but not used in the hot path (app generates).
  - Domain events (per [ADR 0018](0018-domain-model-ddd-aggregates.md)) carry UUIDs as opaque references ‚Äî bounded-context-friendly.

- **Negative:**
  - Index footprint ~2√ó BIGINT. At 5-year scale: `flight_crew` +900 MB, `flight` +450 MB, `audit_log` potentially multi-GB. Total budget ~3-5 GB additional disk + RAM. **Not free; not blocking on a self-hosted VPS.**
  - Joins on UUID columns are ~5-10% slower than BIGINT (16-byte vs 8-byte comparisons, larger working set in shared_buffers). Measurable at very high concurrency; invisible at FLS-scale.
  - URL length: every `/api/v1/flights/{id}` path is 36 chars longer than with BIGINT. Acceptable.
  - Postgres needs the `pgcrypto` extension only if `gen_random_uuid()` is used (it's not ‚Äî app generates). No extension required for the `uuid` column type itself.
  - Legacy data correlation via audit log: a 2024 audit row referencing a Flight by its legacy GUID won't directly match the new system's UUID v7 unless the remapping is preserved (see Open Q above). Handled by S-027 + S-016 coordination.
  - Reviewer guidance: a one-off `BIGINT` PK on a new table is a red flag ‚Äî code review enforces the convention.
