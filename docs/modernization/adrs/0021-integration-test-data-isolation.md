# 0021 — Integration-test data isolation strategy

- **Status:** Accepted
- **Date:** 2026-05-16
- **Decision criteria** (from [vision §6](../02-vision-and-constraints.md#6-decision-criteria-for-phase-3)):
  2. Team-familiar stack
  5. Structural multi-tenancy supported
  6. Preserves sacred cows cleanly
  7. Solo-operator operability
  8. Enables fast feature dev post-cutover

## Context

S-012 retired the H2-with-`MODE=PostgreSQL` fallback and pinned every `@SpringBootTest` to a JVM-singleton Postgres container ([CONVENTIONS.md "Test infrastructure for DB-touching tests"](../../next/server/CONVENTIONS.md)). One container, one schema, every test boots into the same database. S-012's tests are read-only (information_schema queries, reference-seed assertions) so test-data interference hasn't surfaced — but the moment S-022 lands JPA entities and tests start `INSERT`ing Persons / Clubs / Flights, two tests writing to the same shared schema collide.

The legacy e2e suite has had a written "self-contained-parallel" policy since the Playwright suite became parallel-safe ([`flsweb` TEST_WRITING.md §1, mentioned in `e2e/playwright.config.ts:34-37`](../../e2e/playwright.config.ts)). The new-stack Java integration tests need an analogous rule — pinned **before** S-022 sets a precedent we drift from.

The decision space is shaped by three forces:

1. **`@TenantId` is the production isolation mechanism** ([ADR 0008](0008-multi-tenancy-mechanism.md)). Tests should exercise the same mechanism, not bypass it. A test policy that piggybacks on `@TenantId` validates the production constraint by construction.
2. **Cross-tenant entities exist by design** (Person, Location — sacred cows per [00-seed.md](../00-seed.md)). They can't be isolated by `club_id` because they don't carry one. The policy needs a second rule for the cross-tenant case.
3. **Solo-operator workflow** (criterion 7). Developer needs to be able to re-run a single failing test, inspect leftover state in pgAdmin / psql, fix the production code, re-run again — without manually resetting the database between attempts.

## Options considered

### Option A — Per-test unique `club_id` + stable-ID for cross-tenant, pre-clean at start ← chosen

- **Capabilities:**
  - **Tenant-scoped data** (every TENANT_SCOPED + internal-aggregate table): each test creates a unique `Club` row at start (UUID v7 generated by the app, prefix `clb_`); all writes carry that `club_id`; `@TenantId` enforces isolation structurally. Teardown is **not required** — leaving the Club + cascade-children in place lets the developer inspect leftover state. The **next run of the same test** pre-cleans its own data (lookup by the test's stable Club key) and re-creates. Tests against `member_state` / `person_category` / `email_template` etc. are all internal to the test's Club, so cleanup is one `DELETE FROM club WHERE id = ?` with CASCADE.
  - **Cross-tenant data** (Person, Location, and any future cross-tenant entity): each test owns its rows by a **stable, randomized natural key**: a test naming itself `IT_PersonRosterTest_canEditCrew` writes `Person.lastname = 'IT_PersonRosterTest_canEditCrew_<uniq>'`, pre-cleans by that lastname at start. Same self-contained-parallel pattern as the legacy e2e suite (`TEST_WRITING.md §1`).
  - **`@WebMvcTest` slice tests** need no isolation — they don't load a DataSource.
  - **Pure unit tests** need no isolation — they don't touch the database.
  - **Concurrent execution.** Each test's data is keyed by its own `club_id` (tenant-scoped) or stable randomized natural key (cross-tenant). Two tests running in parallel can't see each other's data via the application's normal `@TenantId`-filtered queries; cross-tenant tests don't race because their natural keys are unique by construction. The policy is parallel-safe **by default**.
  - **Per-test pre-clean, not teardown.** A failing test leaves its data in the database. Developer opens pgAdmin (per `next/ops/dev-up-full.sh`) at any time and inspects the exact state the test produced. Re-running the test pre-cleans its own data and starts fresh. No `@AfterEach`, no global TRUNCATE.
- **Fit to criteria:**
  - **2 ✓** — JUnit 5 idioms; one helper `IntegrationTestSupport.createTestClub(label)` plus a stable-ID naming convention. No new framework, no custom JUnit Extension required.
  - **5 ✓** — `@TenantId` IS the isolation mechanism in production; tests exercise it. The test suite validates the production constraint structurally.
  - **6 ✓** — Cross-tenant sacred cows (Person, Location) are honored by the second rule, not shoehorned into a `club_id` they shouldn't have.
  - **7 ✓** — Failed-test state is preserved for inspection; re-runs are self-cleaning. The developer's loop is `gradle test → inspect in pgAdmin → fix code → re-run`. No DB reset step.
  - **8 ✓** — New stories don't pay a setup tax. Each test names its Club / natural keys; the helper handles the rest.
- **Migration cost:** zero — S-012's existing tests are read-only and don't need a Club. The pattern lands the moment S-022 needs it. CONVENTIONS.md grows by ~30 lines; an `IntegrationTestSupport` helper class lands at S-022.
- **Ecosystem risk:** low. Pattern is well-documented in the DDD-on-JPA literature (Khorikov ch. 9) and matches the legacy e2e suite's proven pattern.
- **Escape hatch:** a test that genuinely cannot be tenant-scoped (operator-level cross-club rollup; OGN ingest service principal; cutover script integration) opts out and uses TRUNCATE for the duration of its own setUp. Documented per-test as an exception.

### Option B — Transactional rollback (`@Transactional` + auto-rollback)

- **Capabilities:** Spring's test-context auto-rolls-back any `@Transactional`-annotated test. Zero per-test cleanup; resetting is automatic. Works at the service-layer (in-process method calls within the test's transaction).
- **Fit to criteria:** **5 ✗ — does not survive an HTTP boundary.** A test that uses `TestRestTemplate` or `MockMvc` to hit a controller spans two transactions (the test's outer one, the controller's request-scoped one). The rollback only covers the outer one; data written inside the controller's transaction persists across tests. Multi-tenant integration tests almost always cross the HTTP boundary, so the rule fails on the most important slice.
- **Why not chosen:** the rule has to apply differently to service-layer vs HTTP-layer tests; the bifurcation is brittle. Tests using both patterns in the same class would silently leak data. The cognitive overhead defeats the rule.

### Option C — Per-test Postgres schema (search_path swap per test)

- **Capabilities:** each test gets its own `test_$uuid` schema; Flyway runs migrations into it; teardown drops the schema.
- **Fit to criteria:** **8 ✗ — slow.** V2 is 794 lines, 19 tables, ~250 reference-data INSERTs. Per-test migration adds ~1s + connection-pool reset. At hundreds of tests, that's minutes per CI run.
- **Why not chosen:** the schema-per-test pattern does NOT match the production shape (production uses one shared schema with tenant-by-row), so the tests no longer validate the production constraint. Plus the per-test migration overhead is genuinely costly.

### Option D — TRUNCATE between tests (`@AfterEach` global wipe)

- **Capabilities:** `@AfterEach` issues `TRUNCATE` on every domain table `CASCADE` + resets sequences. Brute-force; no thinking required.
- **Fit to criteria:** **5 ~** — doesn't exercise `@TenantId`; isolation is by wipe, not by tenant. **7 ✗ — failed-test state is wiped before the developer can inspect it.** **8 ✗ — blocks parallel execution** (a `TRUNCATE` mid-suite would also clear another test's in-progress data).
- **Why not chosen:** the inspection-friendliness loss is real for solo-operator workflow, and the parallel-execution lock-out is exactly the wrong direction.

### Option E — Self-contained-parallel only (no per-test Club)

- **Capabilities:** matches the legacy e2e suite exactly. Every test names its rows by stable natural key (across all tables); pre-cleans at start.
- **Fit to criteria:** **5 ~** — tenant-scoped tables would need every test to invent a unique-ID convention per table, with no help from `@TenantId`. **6 ~** — works for cross-tenant but adds friction for tenant-scoped tests (every Flight needs a stable test-owned ID, vs just inheriting from a test-owned Club).
- **Why not chosen alone:** the tenant-scoped case is most of the suite. Letting `@TenantId` do the isolation for tenant-scoped tests is structurally cleaner. Option E is retained as the **second rule** for cross-tenant entities under Option A.

## Decision

Chosen: **Option A — per-test unique `club_id` for tenant-scoped data, stable-randomized natural key for cross-tenant data, pre-clean at test start (not teardown at end).**

### Decision rule (applies from this ADR forward)

1. **Tenant-scoped test data** — each integration test creates a unique `Club` row at start. The Club's name / club_key is derived from the test's class name + method name (stable across re-runs of the same test, unique across tests). All writes carry the test's `club_id`. `@TenantId` filtering keeps the test's data invisible to other tests' queries. Pre-clean at start: `DELETE FROM club WHERE club_key = ?` with CASCADE removes everything the previous run left.

2. **Cross-tenant test data** (Person, Location, anything carrying no `club_id`) — tests own rows by stable randomized natural key (e.g., `Person.lastname = 'IT_<ClassName>_<methodName>_<uniq>'`). Pre-clean at start by that key. Matches the legacy e2e suite's `TEST_WRITING.md §1` self-contained-parallel pattern.

3. **`@WebMvcTest` slice tests** — need no isolation. They don't load a DataSource.

4. **Pure unit tests** — need no isolation. They don't touch the database.

5. **Parallel execution allowed and encouraged.** A test that explicitly cannot run in parallel (operator-level cross-club rollup; OGN ingest service principal; cutover script integration) declares it via `@Execution(SAME_THREAD)` (or by being placed in a single-threaded test set) and documents why in a comment.

6. **Pre-clean, never teardown.** No `@AfterEach`, no global TRUNCATE, no `@Transactional` auto-rollback. Failed-test state stays in the database for the developer to inspect via pgAdmin (`next/ops/dev-up-full.sh`). Re-running the same test pre-cleans its own data.

7. **Helper:** `IntegrationTestSupport.createTestClub(testClassName, testMethodName)` lands with S-022, returns a `Club` aggregate root with a deterministic `club_key`. Pre-cleanup is one method invocation.

## Consequences

- **Positive:**
  - `@TenantId` is exercised by every multi-tenant test — production constraint validated structurally.
  - Parallel execution by default. The Gradle test parallelism + JUnit 5 `@Execution(CONCURRENT)` can both be enabled without rewriting tests.
  - Failed-test inspection is one pgAdmin click away. Developer doesn't need to re-run with a debugger to see the data that produced the failure.
  - Re-running the same test multiple times produces deterministic state because pre-clean is keyed by the test's own name.
  - Cross-tenant sacred cows (Person, Location) keep their cross-tenant shape; tests don't fake a `club_id` they shouldn't have.
  - No `@Transactional` boundary issues. The rule applies uniformly across service-layer and HTTP-layer tests.

- **Negative:**
  - Test author has to choose a natural key for cross-tenant entities — discipline tax. Mitigation: helper that generates the key from `getClass().getSimpleName() + "_" + testMethodName`.
  - The shared schema accumulates test data over time. Mitigation: a `gradle cleanTestData` task that drops every test Club + every row with an `IT_` natural-key prefix. Runs at developer's discretion; not part of normal test flow.
  - Tests that genuinely need a clean slate (cutover script integration, OGN ingest E2E) opt out and pay the TRUNCATE tax. Documented per-case.
  - Slightly slower test setup than `@Transactional` rollback (an INSERT + cleanup at start vs nothing). Negligible at FLS scale; ~5-10ms per test.
