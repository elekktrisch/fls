# 0005 — API shape (REST + OpenAPI + generated TS client)

- **Status:** Accepted
- **Date:** 2026-05-14
- **Decision criteria** (from [vision §6](../02-vision-and-constraints.md#6-decision-criteria-for-phase-3)): preserves sacred cows (Proffix integration) · enables fast feature dev (type-sharing) · mature ecosystem · solo-operator operability

## Context

Two constraints box this decision in:

1. **Sacred cow — Proffix integration.** The external `PROFFIX-FLS-Sync` consumer pulls deliveries from `/api/v1/deliveries/*` over REST (per [seed](../00-seed.md) and [current-state §4](../01-current-state.md#4-integration-map)). The new system has to serve a Proffix-compatible API regardless of what we do for the SPA. The investigation flagged in [vision §8](../02-vision-and-constraints.md#8-open-items-to-resolve-before--during-phase-3) doesn't change this — wherever the live code lives, it speaks REST.
2. **R5 — enum drift between server and client.** `FlightStateMapper` hand-encodes server enum values as JS strings ([current-state §7 R5](../01-current-state.md#r5--flightstatemapper-enum-duplication)). Any drift is a silent UI bug. The replacement needs a generated client whose types come from the server's actual contract, not from hand-edited mirrors.

Together these constraints make the shape "REST with a machine-readable contract" the obvious starting point. The choice is the *kind* of contract and *which* generator.

## Options considered

### Option A — REST + OpenAPI (via springdoc-openapi) + generated TS client
- **Capabilities:** springdoc-openapi-starter-webmvc-ui scrapes Spring controllers + Jakarta validation annotations at startup, emits an OpenAPI 3.1 spec at `/v3/api-docs`. A frontend codegen tool (orval, openapi-typescript / hey-api/openapi-ts, openapi-typescript-codegen) consumes the spec and emits TypeScript types + a typed client. Spec can be served live or snapshot-committed into the frontend repo for reproducible builds.
- **Fit to criteria:** preserves Proffix ✓ (same API shape and conventions as today, just under a new server). Type-sharing ✓ (closes R5 entirely). Mature ecosystem ✓ (springdoc + orval are both well-supported defaults). Solo-operator operability ✓ (one command regenerates types after a backend change).
- **Migration cost:** low. Controllers in [ADR 0001](0001-backend-language-and-framework.md) Spring Boot map idiomatically from existing ASP.NET Web API controllers. Annotation work to fill in `@Operation` / `@Schema` is incremental.
- **Ecosystem risk:** low. OpenAPI is the de facto REST contract standard.
- **Escape hatch:** if springdoc proves limiting, the OpenAPI spec can be hand-written or generated by other tools (smallrye-openapi, etc.); the codegen step is independent of the spec source. REST itself is the lingua franca — replaceable by adding GraphQL or RPC on top later.

### Option B — GraphQL
- **Capabilities:** single endpoint, client-driven querying, strongly-typed schema, real-time subscriptions if needed.
- **Fit to criteria:** Proffix integration ✗ — Proffix sync expects REST; supporting it would require a parallel REST facade or rebuilding Proffix sync (out of scope). Type-sharing ✓. Solo-operator operability ✗ — operating both REST and GraphQL servers is more surface for a solo dev.
- **Migration cost:** high — every CRUD endpoint reshapes into a query/mutation; resolvers replace controllers.
- **Ecosystem risk:** low — Spring for GraphQL is solid; but using GraphQL for an internal admin app is over-engineering.
- **Escape hatch:** REST layer can sit alongside, but at that point we'd be maintaining two contracts.

### Option C — REST + hand-written TypeScript types
- **Capabilities:** lower setup overhead; no codegen step in the build.
- **Fit to criteria:** Type-sharing ✗ — this is exactly [R5](../01-current-state.md#r5--flightstatemapper-enum-duplication). Drift is invisible until a user sees the bug.
- **Migration cost:** lowest initially; pays back negatively over time.
- **Rejected** — would re-introduce the very class of bug we want eliminated.

### Option D — JSON-RPC / custom RPC over HTTP
- **Capabilities:** simpler than REST in some ways; less idiomatic for CRUD.
- **Fit to criteria:** Proffix integration ✗, mature ecosystem ✗ (less tooling).
- **Rejected up front.**

## Decision

Chosen: **Option A — REST + OpenAPI (springdoc) + generated TypeScript client**. The Proffix sacred cow forces REST; OpenAPI + codegen turns that constraint into a feature by giving us a single source of truth for DTOs and enums. The specific codegen library (orval / openapi-typescript / hey-api/openapi-ts) is a phase-4 story, not an ADR — they are interchangeable.

## Consequences

- **Positive:**
  - Proffix sync continues to work against the new API with no changes on its side (subject to the §8 investigation confirming the contract).
  - Closes R5 structurally — frontend can't drift from backend types without a build break.
  - springdoc emits Swagger UI at `/swagger-ui` for free, useful in dev and as ad-hoc documentation.
  - REST is the boring, universally understood interface — no exotic-tooling tax on solo operations.
  - Versioning path remains the familiar URL-prefix pattern (`/api/v1/`, `/api/v2/`) if and when needed.

- **Negative:**
  - OpenAPI codegen quality varies by tool; ugly generated TS can leak into the frontend if we don't pick a good generator. Mitigation: evaluate orval / hey-api / openapi-typescript output during the frontend scaffold story.
  - The OpenAPI spec must be kept accurate. Discipline required: every new endpoint and DTO gets a complete `@Schema` description, validation annotations, response types. Drift between the live API and the generated spec is the new failure mode.
  - GraphQL-style "fetch exactly the fields I need" is unavailable — REST endpoints return full DTOs. Mitigate with thoughtful endpoint design (separate list-view DTOs from detail DTOs as the current code already does).

- **Follow-ups (other ADRs / stories implied):**
  - **ADR 0006** (frontend data fetching) — Angular's signal-based resource()/rxResource() APIs are the natural pairing with a generated REST client; see [ADR 0004](0004-frontend-framework-and-build-tool.md).
  - **ADR 0008** (multi-tenancy) — controllers will use a tenant context injected by Spring Security; the OpenAPI spec must not leak tenant IDs into request bodies/paths where the principal already implies them.
  - **Story:** spike the springdoc setup on a one-endpoint Spring Boot project; verify the generated spec is clean enough for orval / hey-api.
  - **Story:** evaluate 2–3 TS codegen tools against a sample DTO graph (including discriminated unions for `FlightAircraftType` and the `FlightProcessState` enum); pick one.
  - **Story:** establish the spec-publication mechanism — live at `/v3/api-docs` for dev; committed snapshot in `alpenflight/web/` for reproducible CI builds.
  - **Story:** capture the current REST surface (~46 controllers per [current-state §4](../01-current-state.md#4-integration-map)) as the parity baseline for the new system's OpenAPI spec, especially the `/api/v1/deliveries/*` shape Proffix consumes.
